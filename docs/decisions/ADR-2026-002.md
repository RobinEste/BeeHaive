# ADR-2026-002: RAG pipeline — RAG-Anything + vLLM-MLX + Neo4j workspace-isolatie

> **Datum:** 2026-03-05
> **Status:** geaccepteerd

## Context

BeeHaive heeft een knowledge graph (Neo4j) met BuildingBlocks, Guardrails en KnowledgeItems. De volgende stap is Retrieval-Augmented Generation (RAG): documenten opnemen, embeddings genereren, en vragen beantwoorden door graph traversal + vector similarity te combineren met een LLM.

Constraints:
- EU data residency verplicht — geen klantdata naar externe services
- RAG-Anything is onderdeel van een training (Multi-Agent Course Module 4)
- Hergebruik gewenst — dezelfde RAG-stack wordt ook in andere projecten ingezet
- Lokaal development op Apple Silicon Mac (32 GB RAM)
- Meertalig: zowel Engelse als Nederlandse content

## Beslissing

### RAG engine: RAG-Anything (op LightRAG)

RAG-Anything als high-level orchestrator, met LightRAG als onderliggende engine.

- **RAG-Anything** biedt multimodale documentverwerking (PDF, video, afbeeldingen)
- **LightRAG** levert graph-based hybrid retrieval (vector + graph traversal)
- Beide zijn open-source en volledig self-hostable

### LLM + Embeddings: vLLM-MLX (lokaal)

Alle AI-inference draait lokaal via vLLM-MLX op Apple Silicon:

| Component | Model | RAM |
|-----------|-------|-----|
| LLM (entity extraction) | Qwen3.5-9B-4bit | ~6 GB |
| Embeddings | multilingual-e5-large-mlx | ~1.5 GB |

vLLM-MLX serveert beide modellen op één OpenAI-compatible endpoint (`/v1`).

### Graph storage: Neo4j met workspace-isolatie

LightRAG maakt een eigen graph schema aan dat niet compatibel is met het bestaande BeeHaive schema (BuildingBlocks, Guardrails). Oplossing: **workspace-isolatie**.

- Bestaande BeeHaive-graaf blijft ongewijzigd
- LightRAG krijgt een eigen workspace (`beehaive`) binnen dezelfde Neo4j instance
- LightRAG's nodes krijgen het workspace-label als extra label
- Later: bridge-queries die beide schema's verbinden

## Rationale

### RAG-Anything vs. LightRAG direct

LightRAG direct is lichter, maar RAG-Anything biedt multimodale parsing die we nodig hebben voor video's en PDF's (YouTube talks, EU AI Act documenten). Bovendien is het onderdeel van de cursus.

### vLLM-MLX vs. Ollama vs. cloud API

- **Ollama:** simpelere setup, maar minder controle over model serving
- **Cloud API (Voyage, OpenAI):** betere embeddings, maar data gaat buiten EU
- **vLLM-MLX:** volledige controle, EU data residency, en voorbereiding op Hetzner-deployment

### Qwen 3.5 9B vs. grotere modellen

Oorspronkelijk gekozen voor Qwen 3.5 35B-A3B (MoE, ~20 GB RAM), maar dit bleek te zwaar voor lokale development op 32 GB RAM. Qwen 3.5 9B-4bit (~6 GB) biedt voldoende kwaliteit voor entity extraction en graph building, met ruimte over voor andere processen.

### multilingual-e5-large vs. bge-m3

- **bge-m3:** betere benchmark scores, maar geen MLX-variant beschikbaar
- **multilingual-e5-large:** meertalig (incl. NL), beschikbaar als MLX-variant, past binnen de vLLM-MLX setup

## Alternatieven overwogen

### Alternatief 1: Alleen full-text search in Neo4j (geen vector embeddings)

- **Voordelen:** Geen extra infra nodig, simpeler
- **Nadelen:** Geen semantische search, geen RAG, mist de cursus-opdracht
- **Waarom niet gekozen:** Semantische search is essentieel voor de kennisbank

### Alternatief 2: LangChain + ChromaDB

- **Voordelen:** Groot ecosystem, veel tutorials
- **Nadelen:** Geen native graph-based retrieval, extra vector store naast Neo4j
- **Waarom niet gekozen:** LightRAG biedt graph-based hybrid retrieval dat beter past bij onze knowledge graph architectuur

### Alternatief 3: Lokale embeddings via sentence-transformers

- **Voordelen:** Lichtgewicht, geen server nodig
- **Nadelen:** Geen OpenAI-compatible API, lastig te hergebruiken, geen serving layer
- **Waarom niet gekozen:** vLLM-MLX biedt een serving layer die direct herbruikbaar is voor andere projecten

## Consequenties

### Positief
- Volledig self-hosted: geen data buiten de eigen machine/server
- Herbruikbare stack voor meerdere projecten
- Multimodale content verwerking (video, PDF, afbeeldingen)
- Graph-based retrieval combineert vector similarity met graph traversal

### Negatief
- vLLM-MLX + Qwen 3.5 9B vergt ~8 GB RAM (embedding model incl.)
- RAG-Anything heeft zware dependencies (MinerU, LibreOffice voor docs)
- Twee graph schema's in dezelfde Neo4j instance vereist aandacht bij queries

### Risico's
- vLLM-MLX is relatief nieuw — minder mature dan Ollama
- Qwen 3.5 9B prestatie op entity extraction is nog niet gevalideerd
- Workspace-isolatie in Neo4j moet nog bewezen worden met LightRAG

### Mitigatie
- Bij performance-problemen: fallback naar Qwen 3.5 4B
- Bij vLLM-MLX issues: Ollama als alternatief (bge-m3 + Qwen via Ollama)
- Bridge-queries pas bouwen als beide schema's stabiel zijn
