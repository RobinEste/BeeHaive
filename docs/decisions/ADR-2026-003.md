# ADR-2026-003: Externe LLM API's in plaats van lokaal LLM

> **Datum:** 2026-03-09
> **Status:** geaccepteerd
> **Vervangt:** ADR-2026-002 (gedeeltelijk — LLM + embedding keuze)

## Context

ADR-2026-002 koos voor vLLM-MLX + Qwen 3.5 9B lokaal als LLM en multilingual-e5-large als embedding model. Na meerdere sessies bleek de lokale LLM-stack instabiel: OOM errors, quantization-artefacten, model versie mismatches, en vLLM-MLX incompatibiliteiten.

BeeHaive verwerkt uitsluitend publiek beschikbare academische bronnen. PII (contactgegevens) wordt geredacteerd vóórdat tekst naar een externe API gaat (AVG Art. 5(1)(c), zie KNW-2026-008). De EU data residency eis uit ADR-2026-002 gold voor klantdata — de knowledge graph bevat geen klantdata.

De huidige use cases voor LLM:
1. **Ingestion taxonomy mapping:** tekst → BuildingBlocks, Guardrails, Topics, Authors
2. **RAG query answering:** vraag + retrieved context → antwoord
3. **Embeddings:** tekst → vector voor similarity search

## Beslissing

### Lokaal LLM vervalt volledig

vLLM-MLX en alle lokale modellen (Qwen, multilingual-e5-large) worden niet meer gebruikt. Alle LLM-inference gaat via externe API's.

### Ingestion: Gemini 2.5 Flash

De taxonomy mapper gebruikt Gemini 2.5 Flash via Google AI Studio (`google-genai` SDK). Gemini is gekozen omdat:
- Robin heeft een bestaand Google-abonnement
- Flash is snel en goedkoop voor structured output
- Google AI Studio heeft een ruime gratis tier

### RAG queries: nog open

Het LLM voor RAG query answering is nog niet gekozen. Kandidaten: Gemini, Claude, of een ander frontier model. Dit wordt besloten wanneer de RAG query pipeline aangepast wordt aan externe LLM's.

### Embeddings: nog open

Het embedding model moet extern worden. Kandidaten:
- Google text-embedding (via dezelfde Gemini API)
- Voyage AI (goede meertalige embeddings)
- OpenAI text-embedding-3

Dit wordt besloten in samenhang met de RAG query LLM-keuze.

### RAG-Anything/LightRAG: blijft

De RAG engine (RAG-Anything op LightRAG) blijft. De engine ondersteunt externe LLM/embedding endpoints via OpenAI-compatible API. De configuratie wijzigt van lokale vLLM-MLX URL naar de gekozen externe endpoints.

### Neo4j workspace-isolatie: blijft

Ongewijzigd ten opzichte van ADR-2026-002.

## Rationale

### Waarom niet meer lokaal

| Aspect | Lokaal (vLLM-MLX) | Extern (API) |
|--------|-------------------|--------------|
| Stabiliteit | Instabiel (OOM, versie-issues) | Betrouwbaar |
| Kwaliteit | 9B quantized — matig | Frontier — hoog |
| Infra-overhead | Hoog (GPU, model mgmt) | Geen |
| Kosten | Gratis (na hardware) | Laag (free tier/pay-per-use) |
| Privacy | Data blijft lokaal | OK — tekst is PII-geredacteerd |
| Latency | ~2-5s lokaal | ~1-3s via API |

De doorslag: BeeHaive is een CLI-first tool voor één developer. Het volume is laag (<100 calls/dag). De tekst is al geredacteerd. De infra-overhead van lokaal weegt niet op.

### Waarom Gemini als eerste keuze

Pragmatisch: bestaand abonnement, ruime free tier, goede structured output. Geen sterke lock-in — de taxonomy mapper is een enkele functie die makkelijk naar een ander model te switchen is.

## Alternatieven overwogen

### Alternatief 1: Lokaal LLM fixen (Ollama i.p.v. vLLM-MLX)

- **Voordelen:** Data blijft lokaal, geen API-kosten
- **Nadelen:** Zelfde kwaliteitsprobleem (kleine modellen), nog steeds infra-overhead
- **Waarom niet gekozen:** Het probleem is niet alleen vLLM-MLX, het is dat 9B modellen onvoldoende kwaliteit leveren voor betrouwbare classificatie

### Alternatief 2: Claude API voor alles

- **Voordelen:** Beste kwaliteit, vertrouwde tooling
- **Nadelen:** Duurder dan Gemini Flash, geen bestaand abonnement met API-toegang
- **Waarom niet gekozen:** Gemini is goedkoper voor de classificatietaak. Claude blijft een optie voor RAG queries waar kwaliteit belangrijker is.

## Consequenties

### Positief
- Geen lokale GPU/model management meer
- Betere classificatiekwaliteit (frontier vs. 9B quantized)
- Simpelere development setup (geen vLLM-MLX opstarten)
- LLM-keuze per use case optimaliseerbaar (Gemini voor classificatie, mogelijk Claude voor queries)

### Negatief
- Afhankelijkheid van externe API's (availability, rate limits)
- Kosten bij hoog volume (nu verwaarloosbaar)
- `backend/app/rag/` configuratie moet aangepast worden voor externe endpoints

### Risico's
- API downtime blokkeert ingestion — mitigatie: graceful degradation (KnowledgeItem zonder relaties)
- Embedding model switch kan RAG retrieval kwaliteit beïnvloeden — mitigatie: testen op bestaande 34 items
- Vendor lock-in — mitigatie: abstractielaag, modellen zijn verwisselbaar
