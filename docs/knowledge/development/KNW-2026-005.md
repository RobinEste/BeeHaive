# KNW-2026-005: Backwards-compatible aliases bij data refactoring

> **Categorie:** development
> **Datum:** 2026-03-08
> **Sessie-context:** Knowledge graph uitbreiden van 1 naar 34 seed items (Stap 6)
> **Relevantie:** midden

## Inzicht

Bij het herstructureren van data-exports (bijv. single dict → lijst van dicts) kun je backwards-compatible aliases behouden om bestaande imports niet te breken. Dit voorkomt ImportErrors in tests en andere consumers zonder dat je de tests zelf hoeft aan te passen.

## Context

`seed.py` exporteerde oorspronkelijk `EXAMPLE_KNOWLEDGE_ITEM` en `EXAMPLE_RELATIONS` als losse dicts voor één item. Bij de uitbreiding naar 34 items werd de structuur veranderd naar een `KNOWLEDGE_ITEMS` lijst. De test suite importeerde de oude namen direct.

## Geleerd

### Wat werkte
- Aliases toevoegen die naar het eerste item in de nieuwe lijst wijzen: `EXAMPLE_KNOWLEDGE_ITEM = KNOWLEDGE_ITEMS[0]`
- Tests blijven groen zonder aanpassingen

### Wat niet werkte
- Initieel werden de oude exports simpelweg verwijderd → `ImportError` bij test collection

### Waarom
Python imports zijn harde contracten. Zolang consumers bestaan die een naam importeren, moet die naam beschikbaar blijven of moeten alle consumers tegelijk worden bijgewerkt. Bij seed data is een alias goedkoper dan alle tests refactoren.

## Toepassing

Bij elke refactoring van geëxporteerde data-structuren: controleer eerst welke modules de oude namen importeren (`grep -r "from.*import.*NAAM"`). Voeg aliases toe als de oude namen nog in gebruik zijn. Markeer ze optioneel met een comment `# backwards-compat alias` zodat ze later opgeruimd kunnen worden.
