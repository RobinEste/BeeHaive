# KNW-2026-016: Goedkope checks vóór dure I/O in pipelines

> **Categorie:** development
> **Datum:** 2026-03-17
> **Sessie-context:** Ingestion pipeline — deduplicatie check stond na de HTTP fetch in plaats van ervoor
> **Relevantie:** midden

## Inzicht

Plaats goedkope validatie- en deduplicatie-checks altijd vóór dure operaties (netwerk I/O, API calls, CPU-intensief werk). Een database lookup (ms) die een onnodige HTTP fetch (5-15s) voorkomt, is altijd de juiste volgorde. Dit klinkt vanzelfsprekend maar is makkelijk te missen bij refactoring.

## Context

Bij het herstructureren van de pipeline werd de stappenvolgorde per ongeluk gewijzigd: fetch → PII scan → dedup check. De /review performance agent detecteerde dit als regressie. In de vorige versie stond dedup check correct vóór de fetch.

## Geleerd

### Wat werkte
- /review detecteerde de regressie automatisch (performance-reviewer agent)
- Fix was simpel: blok verplaatsen, geen logica-wijziging nodig

### Wat niet werkte
- Bij refactoring van pipeline stappen is het makkelijk om de volgorde te breken
- Unit tests testten individuele stappen, niet de volgorde (mock-based tests verbergen dit)

### Waarom
Bij refactoring focus je op correctheid ("doet elke stap het juiste?"), niet op efficiëntie ("staat elke stap op de juiste plek?"). De pipeline flow is een apart concern dat bewust gereviewed moet worden.

## Toepassing

Pipeline-ontwerp vuistregel:
1. Validatie (input format, types) — gratis
2. Deduplicatie (database lookup) — goedkoop
3. Fetch/download (netwerk I/O) — duur
4. Processing (CPU) — duur
5. Classificatie (API call) — duur + kosten
6. Persistentie (database write) — duur

Bij refactoring: check expliciet of de stappenvolgorde nog klopt.
