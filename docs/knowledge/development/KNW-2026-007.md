# KNW-2026-007: Quick spike vóór pipeline-bouw om sunk cost te voorkomen

> **Categorie:** development
> **Datum:** 2026-03-08
> **Sessie-context:** Plan-review voor ingestion pipeline die zwaar leunde op LightRAG entity extraction
> **Relevantie:** hoog

## Inzicht

Wanneer een pipeline zwaar leunt op een externe tool (LLM, RAG framework, API) waarvan de output-kwaliteit onzeker is, voeg dan een "Fase 0: quick spike" toe aan het plan. Test handmatig met 3-5 representatieve items in een REPL vóórdat je pipeline-code schrijft. Dit voorkomt dat je dagen investeert in code die op een ongeschikte tool leunt.

## Context

PLAN-007 had een go/no-go evaluatie gepland na Fase 2 (batch ingest van 34 items + vergelijkingsscript). De Critical Consumer agent signaleerde dat dit te laat is: op dat moment is de fetcher, PII scanner en batch-script al gebouwd. Als LightRAG onvoldoende entities extraheert, is dat werk sunk cost. Door eerst 3-5 items handmatig door LightRAG te sturen (15 minuten werk), kun je dezelfde beslissing nemen zonder code te schrijven.

## Geleerd

### Wat werkte
- Quick spike als Fase 0: geen code, alleen een REPL-sessie
- Concrete go/no-go criteria definiëren vóór de spike ("herkenbare entities die overlappen met de taxonomie")
- Spike-resultaat documenteren als formeel beslismoment in de DoD

### Wat niet werkte
- Go/no-go pas na volledige batch-implementatie — te laat, te veel sunk cost
- Vertrouwen op tooling-documentatie zonder handmatige validatie

### Waarom
De onzekerheid zit in de tool, niet in je eigen code. Je eigen code (fetcher, PII scanner) is deterministisch — die werkt of niet. De tool (LightRAG entity extraction met Qwen 3.5) is probabilistisch en afhankelijk van inputkwaliteit. Test de onzekere component eerst.

## Toepassing

Bij elk plan dat leunt op een externe tool met onzekere output:
1. Identificeer de onzekere component (LLM output, API kwaliteit, scraping resultaat)
2. Voeg een Fase 0 toe: handmatige test met 3-5 items
3. Definieer concrete go/no-go criteria
4. Documenteer het resultaat als formeel beslismoment
5. Bouw pas pipeline-code als de spike positief is
