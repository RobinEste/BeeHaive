# KNW-2026-009: SSRF-preventie checklist voor server-side fetchers

> **Categorie:** security
> **Datum:** 2026-03-09
> **Sessie-context:** Ingestion pipeline document fetcher
> **Relevantie:** hoog

## Inzicht

Elke server-side HTTP-client die URLs van gebruikers of configuratie ophaalt, heeft minimaal vier SSRF-maatregelen nodig: scheme-restrictie, DNS-resolutie check, redirect-blokkering, en rate limiting. Dit geldt ook voor CLI-tools en interne services — niet alleen voor web-facing endpoints.

## Context

De ingestion fetcher haalt URLs op uit een configuratielijst (niet direct user input), maar dezelfde SSRF-patronen zijn toegepast omdat de URLs uit externe bronnen komen en de fetcher op dezelfde machine draait als de database.

## Geleerd

### Wat werkte
- **HTTPS-only**: reject alles behalve `https://` (met localhost-uitzondering voor dev)
- **DNS-resolutie check**: `socket.getaddrinfo()` → check tegen private IP ranges vóór de request
- **Redirect blokkering**: `follow_redirects=False` + handmatige redirect detectie (voorkomt SSRF via redirect naar intern adres)
- **Rate limiting**: 1 req/sec met threading.Lock voor thread-safety

### Wat niet werkte
- DNS rebinding (TOCTOU tussen resolve en request) bleef als known risk open — mitigatie vereist DNS-pinning op library-niveau

### Waarom
SSRF is een van de meest voorkomende kwetsbaarheden bij server-side fetchers. Zelfs "interne" tools die geen direct user input nemen kunnen via supply chain (gecompromitteerde URL-lijsten) of misconfiguratie worden misbruikt.

## Toepassing

Checklist voor elke nieuwe server-side HTTP-client:
- [ ] Scheme-restrictie (HTTPS-only in productie)
- [ ] DNS-resolutie check tegen private IP ranges (10.x, 172.16-31.x, 192.168.x, 127.x, link-local, IPv6 loopback/ULA)
- [ ] Redirects niet automatisch volgen
- [ ] Rate limiting (met thread-safety als async/threaded)
- [ ] Timeout op requests (max 30s)
- [ ] User-Agent header met contactinfo
