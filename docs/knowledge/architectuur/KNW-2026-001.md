# KNW-2026-001: Cartesian product preventie met WITH+collect() in Neo4j

> **Categorie:** architectuur
> **Datum:** 2026-03-04
> **Sessie-context:** Query-laag bouwen voor BeeHaive knowledge graph (13 Cypher functies)
> **Relevantie:** hoog

## Inzicht

Meerdere opeenvolgende OPTIONAL MATCH clauses in Neo4j creëren een cartesian product — elke match vermenigvuldigt het aantal tussenresultaten. Dit leidt tot duplicaten in de output en exponentieel geheugengebruik bij groeiende data.

## Context

`get_item_with_relations()` moest een KnowledgeItem ophalen met alle gerelateerde BuildingBlocks, Guardrails, Topics en Authors. De eerste implementatie gebruikte 4 opeenvolgende OPTIONAL MATCHes zonder tussentijdse aggregatie.

## Geleerd

### Wat werkte
- `WITH ki, collect(DISTINCT bb) AS building_blocks` na elke OPTIONAL MATCH voorkomt het cartesian product
- Elke OPTIONAL MATCH wordt eerst geaggregeerd voordat de volgende begint
- `collect(DISTINCT ...)` zorgt voor unieke resultaten

### Wat niet werkte
- 4 opeenvolgende OPTIONAL MATCHes zonder WITH+collect() gaven duplicaten
- Bij 7 BBs × 7 guardrails × 1 topic × 1 author = 49 rijen i.p.v. 1

### Waarom
Neo4j bouwt intern een "result row" per combinatie van matches. Zonder tussentijdse aggregatie groeit dit multiplicatief. Met WITH+collect() wordt elke stap gereduceerd tot één rij voordat de volgende OPTIONAL MATCH begint.

## Toepassing

Gebruik altijd WITH+collect() tussen opeenvolgende OPTIONAL MATCHes als je meerdere relatie-types van een node wilt ophalen. Patroon:

```cypher
MATCH (n:Node {id: $id})
OPTIONAL MATCH (n)-[:REL_A]->(a)
WITH n, collect(DISTINCT a) AS as_list
OPTIONAL MATCH (n)-[:REL_B]->(b)
WITH n, as_list, collect(DISTINCT b) AS bs_list
RETURN n, as_list, bs_list
```
