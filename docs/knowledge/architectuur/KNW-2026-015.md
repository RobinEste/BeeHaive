# KNW-2026-015: Dual ingest met zichtbare graceful degradation

> **Categorie:** architectuur
> **Datum:** 2026-03-17
> **Sessie-context:** Pipeline schrijft naar Neo4j (primair) en LightRAG (secundair) — dual ingest
> **Relevantie:** hoog

## Inzicht

Bij dual ingest naar meerdere datastores: maak failures in secundaire stores non-fatal maar **zichtbaar**. Stille `try/except` met alleen logging creëert data-inconsistentie die de gebruiker niet kan detecteren. Voeg een expliciet veld toe aan het resultaat (bijv. `rag_synced: bool`) zodat de caller weet of alle stores gesynchroniseerd zijn.

## Context

De ingestion pipeline schrijft eerst naar Neo4j (KnowledgeItem + relaties), daarna naar LightRAG (voor RAG query retrieval). Als LightRAG faalt (bijv. API outage), werd het item als "ingested" gerapporteerd zonder indicatie dat het niet via RAG vindbaar was.

## Geleerd

### Wat werkte
- `rag_synced` boolean veld in IngestionResult — simpel, effectief
- CLI toont waarschuwing bij `rag_synced=False`
- KnowledgeItem is altijd gepersisteerd (primaire store faalt nooit door secundaire store)

### Wat niet werkte
- Alleen loggen van LightRAG failures — gebruiker ziet logs niet bij CLI-gebruik
- `status="ingested"` zonder nuance misleidt over de werkelijke staat

### Waarom
Gebruikers vertrouwen op de status. Als "ingested" betekent "misschien niet volledig gesynchroniseerd", is dat een verborgen faal-modus. Het principe is: **failures mogen non-fatal zijn, maar nooit onzichtbaar**.

## Toepassing

Bij elke multi-store write:
1. Primaire store failure → fatal, return error
2. Secundaire store failure → non-fatal, maar return een veld dat de partiële staat aangeeft
3. CLI/UI toont een waarschuwing bij partiële sync
4. Overweeg een reconciliation-mechanisme voor later herstel
