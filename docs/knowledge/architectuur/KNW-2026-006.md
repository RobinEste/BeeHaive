# KNW-2026-006: Read/write scheiding in database query-modules

> **Categorie:** architectuur
> **Datum:** 2026-03-08
> **Sessie-context:** Plan-review voor ingestion pipeline die write-functies nodig had in een read-only query-module
> **Relevantie:** hoog

## Inzicht

Wanneer een module een expliciete read-only conventie heeft (gedocumenteerd in docstring, alleen `execute_read()`), voeg dan geen write-functies toe aan hetzelfde bestand. Maak een apart `mutations.py` (of `commands.py`) naast het bestaande `queries.py`. Dit behoudt de conventie, maakt write-operaties auditbaar, en voorkomt dat de module twee contracten tegelijk claimt.

## Context

`graph/queries.py` bevatte 13 read-only functies met een expliciete docstring: "All functions are read-only (MATCH/RETURN only)". De ingestion pipeline had write-functies nodig (`create_knowledge_item_with_relations`). Twee onafhankelijke review-agents (Architecture Advisor en Feasibility Checker) signaleerden beide dat het toevoegen van writes aan queries.py de conventie breekt.

## Geleerd

### Wat werkte
- Aparte bestanden per operatie-type: `queries.py` (reads) + `mutations.py` (writes)
- Design constraint documenteren: "Geen Cypher buiten `graph/`" geldt voor beide bestanden
- Bestandsnaam communiceert het contract: je weet bij het openen of je reads of writes ziet

### Wat niet werkte
- Alles in één bestand stoppen "omdat het toch dezelfde database is" — dit verbergt de impact van writes

### Waarom
Write-operaties hebben fundamenteel andere risico's dan reads (data-integriteit, transactie-scope, rollback). Door ze fysiek te scheiden zijn ze makkelijker te auditen, te testen, en te reviewen. Bij een incident kun je snel zien welke code data kan wijzigen.

## Toepassing

Bij elk project met een database query-laag:
- Start met een read-only module (`queries.py`)
- Zodra writes nodig zijn: maak `mutations.py` in dezelfde directory
- Documenteer de scheiding in de module-docstrings
- Dezelfde design constraints (parametrisatie, geen f-strings) gelden voor beide
