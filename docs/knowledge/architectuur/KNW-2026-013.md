# KNW-2026-013: Allowlist-validatie op LLM output voor gestructureerde classificatie

> **Categorie:** architectuur
> **Datum:** 2026-03-11
> **Sessie-context:** Bouwen van Gemini taxonomy mapper die tekst classificeert naar bekende BeeHaive-entiteiten
> **Relevantie:** hoog

## Inzicht

Wanneer een LLM tekst moet classificeren naar een vooraf gedefinieerde set entiteiten, valideer de output altijd tegen een allowlist. Structured output (response_schema) garandeert het juiste format, maar niet dat de waarden kloppen — het model kan namen verzinnen die niet in je taxonomie bestaan. Een server-side allowlist-check is de laatste verdedigingslinie.

## Context

De Gemini taxonomy mapper classificeert tekst naar 7 BuildingBlocks en 7 Guardrails. Hoewel de system prompt de exacte namen bevat en Gemini's structured output het response-format afdwingt, kan het model alsnog een BuildingBlock-naam retourneren die niet in de taxonomie staat. Zonder validatie zou dit foute relaties in de knowledge graph opleveren.

## Geleerd

### Wat werkte
- Structured output (`response_schema`) voor format-garantie (altijd parseerbaar JSON)
- Server-side allowlist-check: `if matched_name not in BUILDING_BLOCKS: return None`
- Confidence threshold (>= 0.7) als extra filter op twijfelgevallen
- Combinatie van drie lagen: prompt instructie + structured output + allowlist

### Wat niet werkte
- Alleen op de prompt vertrouwen — het model volgt instructies meestal maar niet altijd
- Alleen structured output — garandeert format, niet inhoud

### Waarom
LLMs zijn probabilistisch. Zelfs met expliciete instructies en structured output is er een kans op "hallucinated" waarden. Bij classificatietaken met een gesloten set opties is een allowlist goedkoop (O(1) lookup) en elimineert het deze hele klasse fouten.

## Toepassing

Pas dit patroon toe wanneer:
- Een LLM moet kiezen uit een vooraf gedefinieerde set opties
- De output wordt gebruikt om relaties of records aan te maken in een database
- Foute waarden moeilijk te detecteren of terug te draaien zijn

Drie-lagen-defensie:
1. **Prompt:** vermeld de exacte opties in de system prompt
2. **Structured output:** gebruik response_schema/function calling voor format
3. **Allowlist:** valideer waarden server-side tegen de bekende set
