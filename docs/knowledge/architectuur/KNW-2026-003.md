# KNW-2026-003: Efficiënt navigatiepatroon voor gerelateerde items in Neo4j

> **Categorie:** architectuur
> **Datum:** 2026-03-04
> **Sessie-context:** Query-laag bouwen voor BeeHaive knowledge graph (13 Cypher functies)
> **Relevantie:** hoog

## Inzicht

Om gerelateerde items te vinden via gedeelde nodes (bijv. items die dezelfde BuildingBlock, Guardrail of Topic delen), is een "shared node navigation" patroon efficiënter dan EXISTS subqueries met aparte pattern comprehensions. Het voorkomt dubbele traversals en maakt sortering op relevantie (aantal gedeelde nodes) eenvoudig.

## Context

`find_related_items()` moest KnowledgeItems vinden die minstens één BuildingBlock, Guardrail of Topic delen met een gegeven item. De eerste implementatie gebruikte EXISTS subqueries voor filtering en aparte pattern comprehensions voor het ophalen van data — dit traverseerde dezelfde paden twee keer.

## Geleerd

### Wat werkte
- Eén MATCH patroon: `(ki)-[:RELATES_TO|ADDRESSES|ABOUT]->(shared)<-[:RELATES_TO|ADDRESSES|ABOUT]-(other)`
- COUNT(DISTINCT shared) voor relevantie-ranking
- LIMIT 25 voor bounded resultaten
- WHERE clause op het bronitem (titel) en uitsluiting van zichzelf

### Wat niet werkte
- EXISTS subqueries + aparte pattern comprehensions (dubbele traversal)
- Ongelimiteerde resultaten bij groeiende graaf

### Waarom
Het navigation pattern traverseert elke relatie precies één keer. EXISTS + comprehension traverseert dezelfde paden twee keer: eerst voor filtering, dan voor data. Bij een groeiende graaf verdubbelt dit de query-kosten.

## Toepassing

Gebruik het shared-node navigation pattern wanneer je "meer zoals dit" functionaliteit bouwt:

```cypher
MATCH (source {title: $title})-[:REL_A|REL_B|REL_C]->(shared)<-[:REL_A|REL_B|REL_C]-(other)
WHERE other <> source
WITH other, COUNT(DISTINCT shared) AS shared_count
ORDER BY shared_count DESC
LIMIT 25
RETURN other {.*, shared_count: shared_count}
```

Voeg altijd een LIMIT toe — zonder begrenzing groeit het resultaat onvoorspelbaar.
